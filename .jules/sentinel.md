## 2025-10-24 - Missing Security Headers Mitigation
**Vulnerability:** The application was missing standard HTTP security headers globally (such as HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy).
**Learning:** Next.js uses an empty `next.config.js` by default without enforcing strict security headers out of the box. Leaving these headers absent exposes the application to common baseline web vulnerabilities like MIME sniffing, Clickjacking, non-HTTPS downgrades, and overly permissive referrer leaks.
**Prevention:** Always configure an `async headers()` method in `next.config.js` to define fundamental defense-in-depth security headers like `Strict-Transport-Security`, `X-Content-Type-Options`, `X-Frame-Options`, and `Referrer-Policy`.

## 2025-10-24 - Missing Input Bounds Checking
**Vulnerability:** The application was not truncating or sanitizing user input strings (names, addresses, notes) or validating numeric types and bounds correctly before doing `supabase.from(...).insert()` inserts. Because Supabase passes through the input directly without row level security or built-in payload size limits on the proxy layer, this exposes the database to potential abuse, storing extremely large payloads or unhandled malformed data which can cause application disruption and storage bloat.
**Learning:** Client-side components connecting directly to Supabase with an Anon Key must take explicit care to sanitize and truncate raw user input strings to reasonable lengths, and enforce strict type safety/parsing (like using parseInt alongside isNaN checks) before interacting with the database.
**Prevention:** Always validate and sanitize user inputs before any direct-to-database requests. Define maximum lengths for text fields (e.g., `name.trim().slice(0, 100)`) and safely parse strings to their required numeric representations.
